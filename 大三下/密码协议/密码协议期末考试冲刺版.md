## 2. 选择题速记

### 协议和安全协议

协议：两个或两个以上参与者为完成某项特定任务而执行的一系列有序步骤。

协议三层含义：

- 有序过程，每一步依次执行。
- 至少两个参与者。
- 执行后能完成某项任务。

安全协议/密码协议：使用密码算法和密码技术实现安全目标的协议，如认证、密钥建立、隐私保护等。

### 安全协议性质

- 认证性：确认参与方是意定实体。
- 机密性：非授权者不能获得消息内容。
- 完整性：消息不能被非法篡改、删除、替换。
- 不可否认性：事后不能否认发送/接收行为。

注意：DH 被中间人攻击，核心违反的是认证性。

### 消息认证、身份认证、新鲜性

- 消息认证：确认消息来源真实且未被篡改。
- 身份认证：确认通信实体身份。
- 消息认证是身份认证的基础，但二者不是同一概念。
- 新鲜性：消息不是旧消息重放，常用 nonce、时间戳、序列号实现。

### 常用密码工具

- Hash：公开摘要函数，不带密钥时不能单独认证身份。
- MAC/HMAC：带共享密钥，提供消息认证和完整性。
- 数字签名：私钥签名、公钥验证，提供公开验证和不可否认。
- 对称加密：高效，主要提供机密性。
- 公钥密码：用于加密、签名、密钥交换，但传统 RSA/DH/ECC 受量子威胁。

### 攻击手段

至少会写 8 个：

- 窃听攻击。
- 中间人攻击 MITM。
- 重放攻击。
- 假冒攻击。
- 并行会话/交错攻击。
- 未知密钥共享攻击 UKS。
- 离线口令猜测攻击。
- 小子群攻击。
- 降级攻击。
- 反同步攻击 de-synchronization。
- Oracle 攻击，如 password oracle、互素差别 oracle。

### TLS 速记

SSL/TLS 位于应用层和 TCP 之间。

四个子协议：

- Handshake：协商算法、认证、密钥交换。
- Record：分片、压缩、MAC、加密、封装。
- Change Cipher Spec：启用新密码参数。
- Alert：错误告警。

Session vs Connection：

- Session：握手创建的安全参数集合，可被多个连接共享。
- Connection：具体一条临时通信链路，用来传应用数据。

TLS 1.3 为什么需要：

- RSA 密钥传输无前向安全性。
- CBC、RC4、SHA-1 等不再安全。
- 旧版本易受 BEAST、Lucky 13、FREAK、LogJam 等攻击。
- TLS 1.3 简化握手，默认使用具有前向安全性的 DH/ECDHE。

## 3. 七类大题答案模板

### 题 1：什么是协议？什么是安全协议？安全协议有哪些性质？

协议是两个或两个以上参与者为完成某项特定任务而采取的一系列有序步骤。协议要求参与方明确每一步，按约定执行，并能完成预定任务。

安全协议也称密码协议，是利用密码算法和密码机制实现安全目标的协议，例如身份认证、消息认证、密钥建立、隐私保护、安全多方计算等。

安全协议常见性质：

- 认证性：确认对方为意定实体，防止假冒。
- 机密性：保护消息不被非授权者读取。
- 完整性：保护消息不被篡改、删除、替换。
- 不可否认性：保证通信行为事后可证明，不能轻易否认。

可补充：

- 新鲜性：保证消息属于当前协议运行，不是旧消息重放。
- 前向安全性：长期密钥泄露不影响过去会话密钥安全。

### 题 2：针对密码协议有哪些攻击手段？至少 4 种

答题时不要只列名词，要加一句解释：

- 窃听攻击：攻击者监听通信内容，试图获得明文、密钥或认证信息。
- 中间人攻击：攻击者位于双方之间，分别与双方建立连接，使双方误以为在直接通信。
- 重放攻击：攻击者保存旧消息，在之后重新发送，使接收方误以为是当前消息。
- 假冒攻击：攻击者冒充合法参与方执行协议。
- 并行会话/交错攻击：攻击者同时启动多个协议运行，把一个会话中的消息转移到另一个会话使用。
- UKS 攻击：双方共享了同一个密钥，但对密钥归属方的认识不一致。
- 离线口令猜测：攻击者拿到通信记录后，本地枚举口令并验证。
- 小子群攻击：利用群参数检查不足，使对方计算落入小子群，从而泄露秘密信息。
- Oracle 攻击：协议响应泄露可判定信息，攻击者反复查询来筛选秘密。

### 题 3：Schnorr 交互式零知识证明 + Fiat-Shamir

#### 3.1 Schnorr 三轮交互

目标：Alice 向 Bob 证明自己知道私钥 \(x\)，公钥为 \(Y=g^x\)，但不泄露 \(x\)。

流程：

```text
公共参数：群 G，生成元 g，阶 q，公钥 Y=g^x

1. 承诺：
   Alice 选随机数 r <- Z_q
   计算 R = g^r
   Alice -> Bob: R

2. 挑战：
   Bob 选随机挑战 c
   Bob -> Alice: c

3. 响应：
   Alice 计算 z = r + c*x mod q
   Alice -> Bob: z

4. 验证：
   Bob 检查 g^z == R * Y^c
```

为什么正确：

```text
g^z = g^(r+c*x) = g^r * (g^x)^c = R * Y^c
```

三个性质：

- 完备性：诚实 Alice 知道 \(x\)，Bob 会接受。
- 可靠性：不知道 \(x\) 的人难以通过随机挑战。
- 零知识性：Bob 只知道 Alice 知道 \(x\)，不知道 \(x\)。

重要安全点：

如果两次运行复用同一个随机数 \(r\)，私钥会泄露：

```text
z  = r + c*x
z' = r + c'*x
x = (z - z') / (c - c') mod q
```

#### 3.2 Fiat-Shamir 变换

Fiat-Shamir 的作用：把三轮交互式证明压缩成一轮非交互式证明。

核心思想：

- 原来第二轮挑战 \(c\) 由 Bob 随机选择。
- 非交互式版本中，用 Hash 函数生成挑战：

```text
c = H(g, Y, R, message/context)
```

非交互式流程：

```text
Alice:
1. 选 r，算 R=g^r
2. 算 c=H(g,Y,R,上下文)
3. 算 z=r+c*x mod q
4. 输出证明 pi=(R,z)

Verifier:
1. 重新计算 c=H(g,Y,R,上下文)
2. 检查 g^z == R*Y^c
```

答题关键句：

Hash 函数在随机预言机模型 ROM 中充当挑战者，代替交互式协议中验证者选择的随机挑战。

### 题 4：Needham-Schroeder 对称密钥认证协议 NS_v5 的问题与改进

先写背景：

Needham-Schroeder 类协议使用 nonce 挑战-应答机制保证消息新鲜性，并通过可信服务器分发会话密钥。

NS_v5 改进点：

- 进一步将 Alice 的挑战 \(N_A\) 保密。

但 NS_v5 仍存在重放攻击。

攻击条件：

- 攻击者获得了某次旧会话密钥 \(K'_{AB}\)。
- 攻击者保存了旧协议消息。

攻击过程：

```text
1. 攻击者用旧会话密钥 K'_AB 解密旧消息 5，得到 N_A 和 N_B。
2. 攻击者重放旧的 A -> B 消息 3。
3. Bob 根据旧消息继续协议。
4. Bob 认为正在与 Alice 通信，实际上攻击者在利用旧会话材料。
```

攻击结果：

- Bob 接受了旧会话。
- 旧会话密钥可能已经泄露，所以认证和新鲜性失败。

违反性质：

- 新鲜性。
- 认证性。
- 会话密钥的新鲜性/独立性。

改进方向：

- 不能仅加密 nonce，要把 nonce、身份、会话密钥、消息方向和当前运行绑定起来。
- 后续 NS_v6 试图把消息 3、消息 5 改为包含 \(h(N_B),h(N_A)\)，但课件也继续指出其仍有问题。
- 更本质的改进是使用明确的会话标识、双方身份、nonce 和密钥确认机制，并避免接受旧会话密钥相关消息。

可背答案：

> NS_v5 的问题在于旧会话密钥泄露后，攻击者能从旧消息中恢复 nonce，并重放旧消息使 Bob 接受旧会话。根因是协议没有充分把认证材料和当前会话绑定。改进应加入明确身份、nonce、会话标识和密钥确认，避免旧消息在新会话中被接受。

### 题 5：SPEKE 协议

#### 5.1 为什么 \(g=s^2 \bmod p\)

SPEKE 使用安全素数：

```text
p = 2q + 1
```

用口令派生：

```text
g = s^2 mod p
```

平方目的：

- 使 \(g\) 落入阶为 \(q\) 的二次剩余子群。
- 避免生成元落入错误阶或小子群。
- 防止小子群相关攻击。
- 在安全素数群里检查成本低。

答题句：

> 平方是为了把口令映射得到的元素限制到素数阶 \(q\) 的子群中，从而保证后续 DH 运算在正确的大素数阶子群内进行。

#### 5.2 改进后的 SPEKE 流程图

按课件中的改进版写：

```text
公共参数：p, q, p=2q+1
共享口令：pw
令 f(pw)=H(pw)^2，g <- f(pw)

Alice A                                      Bob B
---------------------------------------------------------------
x <- Z_q^*
X = g^x
              A, X  -------------------->

                                      y <- Z_q^*
                                      Y = g^y
              <--------------------  B, Y

检查 Y 是否在合法范围/正确子群
s_A = H(B || X)
s_B = H(A || Y)
sID = max(s_A,s_B) || min(s_A,s_B)
k = KDF(sID || Y^x)

                                      检查 X 是否在合法范围/正确子群
                                      s_A = H(B || X)
                                      s_B = H(A || Y)
                                      sID = max(s_A,s_B) || min(s_A,s_B)
                                      k = KDF(sID || X^y)

可选密钥确认：
H(A||B||g^x||g^y||g^(xy)||g) -------->
<-------- H(B||A||g^y||g^x||g^(xy)||g)
```

最重要的点：

- \(g\) 来自口令。
- 双方通过 DH 得到 \(g^{xy}\)。
- 把身份和临时公钥放入 \(s_A,s_B,sID\)，防止反射、并行和身份绑定问题。
- 密钥确认消息要包含方向、身份、临时公钥、共享 DH 值和生成元。

#### 5.3 认证阶段两条消息能否调换？

按作业讲解答：不能任意调换。

原因：

- 如果先发送的认证值能推出后发送的认证值，例如先暴露 \(H(K)\)，攻击者可直接计算 \(H(H(K))\)，但并不知道会话密钥 \(K\)。
- 这样无法证明参与方真正知道 \(K\)。
- 正确设计应让认证值有方向区分，并绑定身份、临时公钥、会话密钥和 transcript。

可背答案：

> 不能调换。认证消息的顺序和形式用于证明对方确实知道会话密钥。如果先发送的消息可以被直接哈希得到另一条消息，攻击者即使不知道会话密钥也能伪造响应，因此会破坏密钥确认。

#### 5.4 SPEKE 平行会话攻击为什么不是彻底成功？

- 攻击者可利用并行会话通过某些认证检查。
- 但攻击者不能计算最终会话密钥 \(g^{xy}\)。
- 所以它不是完全成功的攻击，只是不完美认证/密钥确认层面的攻击。

### 题 6：HMQV 为什么绑定身份

题目：

HMQV 中：

```text
d = H(X, "Bob")
e = H(Y, "Alice")
```

为什么要把意定通信方身份与密钥派生绑定？如果改成：

```text
d = H(X)
e = H(Y)
```

是否有安全缺陷？

答案：

HMQV 把临时公钥 \(X,Y\) 和通信方身份 Alice/Bob 一起 Hash，是为了把“这个密钥是和谁建立的”放入密钥计算过程。

目的：

- 防止未知密钥共享攻击 UKS。
- 防止身份误绑定。
- 防止攻击者把同一临时公钥转移到不同身份上下文中。
- 确保双方对会话密钥的对端身份认识一致。

如果改成 \(H(X)\)、\(H(Y)\)：

- 身份不再参与密钥派生。
- 攻击者可能让一方认为密钥属于 Bob，另一方认为属于其他人。
- 因此存在安全缺陷。

可背答案：

> 这样处理是为了实现身份绑定和密钥归属绑定，防止 UKS 攻击。若只对临时公钥 Hash，密钥派生不再包含意定通信方身份，攻击者可能造成双方共享同一密钥但对对方身份理解不一致，所以是不安全的。

### 题 7：Shamir \((t,n)\) 门限方案

往届考法：构造一个 \((3,5)\) 门限方案，要求 \(p=17\)，共享密钥 \(k=13\)。

#### 7.1 通用构造

Shamir \((t,n)\)：

- 任意 \(t\) 份 share 可恢复秘密。
- 少于 \(t\) 份不能确定秘密。

构造：

```text
选择素数 p
秘密 k 是多项式常数项
随机选 t-1 个系数
f(x)=a_{t-1}x^{t-1}+...+a_1x+k mod p
给第 i 个用户 share: (i, f(i))
```

恢复：

- 用任意 \(t\) 个点做拉格朗日插值。
- 秘密为 \(f(0)=k\)。

#### 7.2 本题完整例子

要求：

```text
t=3, n=5, p=17, k=13
```

因为 \(t=3\)，选 2 次多项式。若题目没有指定随机系数，可以自己选。这里取：

```text
f(x) = 4x^2 + 3x + 13 mod 17
```

生成 5 个 share：

```text
f(1) = 4 + 3 + 13 = 20 = 3 mod 17
f(2) = 16 + 6 + 13 = 35 = 1 mod 17
f(3) = 36 + 9 + 13 = 58 = 7 mod 17
f(4) = 64 + 12 + 13 = 89 = 4 mod 17
f(5) = 100 + 15 + 13 = 128 = 9 mod 17
```

所以可分发：

```text
(1,3), (2,1), (3,7), (4,4), (5,9)
```

用前三个恢复验证：

拉格朗日系数在 \(x=0\)：

```text
lambda_1 = (0-2)(0-3) / ((1-2)(1-3)) = 6/2 = 3 mod 17
lambda_2 = (0-1)(0-3) / ((2-1)(2-3)) = 3/(-1) = -3 = 14 mod 17
lambda_3 = (0-1)(0-2) / ((3-1)(3-2)) = 2/2 = 1 mod 17
```

恢复：

```text
k = 3*3 + 1*14 + 7*1
  = 9 + 14 + 7
  = 30
  = 13 mod 17
```

所以恢复出秘密 \(k=13\)。

考试注意：

- 系数可随机选，只要多项式次数是 \(t-1=2\)，常数项是 13。
- \(p=17\) 固定时，所有计算都要 mod 17。
- 如果只让“构造”，写出多项式和 5 个 share 通常就够；如果让“恢复”，再写拉格朗日插值。

## 4. 作业高频附加题

### DH 中间人攻击

基础 DH：

```text
Alice -> Bob: g^x
Bob -> Alice: g^y
共享密钥：g^(xy)
```

MITM：

```text
Alice -> Eva: g^x
Eva -> Bob: g^z
Bob -> Eva: g^y
Eva -> Alice: g^z
```

结果：

- Alice 与 Eva 共享 \(g^{xz}\)。
- Bob 与 Eva 共享 \(g^{yz}\)。
- Alice/Bob 都以为和对方通信。

违反认证性。

改进：

- 对 DH 临时值签名。
- 或把身份、公钥、临时值放入密钥派生。

### 先签名后加密 vs 先加密后签名

先签名后加密：

- 优点：提供机密性和认证性，更常用。
- 缺点：不能公开验证，只有接收方解密后能验证；效率略低。

先加密后签名：

- 优点：密文外签名，可公开验证；只加密明文，效率较高。
- 缺点：若明文/签名对象没有绑定发送方身份，攻击者可能替换外层签名。

按作业讲解：实践中更倾向先签名后加密。

### 量子计算影响

- 对称密码：Grover 算法使安全强度约减半，可加长密钥。
- 非对称密码：Shor 算法可破解 RSA、DH、ECC 的整数分解/离散对数难题，影响颠覆性。
- 口令：量子计算不改变口令低熵本质，仍需猜测。
- 口令协议：若依赖传统公钥困难问题，则需抗量子替换或混合协议。

### SRP 三轮优化漏洞

题中优化：

```text
C -> S: I, A
S -> C: s, B, M1
C -> S: M2

M1 = H(A, B, S)
M2 = H(A, M1, S)
```

漏洞：

- 服务器提前发送 \(M_1\)。
- 客户端还没证明知道口令。
- 攻击者可把服务器当 password oracle，离线枚举口令。

攻击：

```text
1. 攻击者发 I,A。
2. 服务器返回 s,B,M1。
3. 攻击者猜 pw'，计算候选 S'。
4. 检查 H(A,B,S') 是否等于 M1。
5. 相等则口令正确。
```

改进：

- 服务器不要提前发送可离线验证的认证值。
- 调整密钥确认顺序，让客户端先证明知道口令。

### RSA-GPAKE 分离攻击

本质：

- 服务器响应提供了“互素差别 oracle”。
- 攻击者选择特殊模数，利用服务器响应排除口令。

若 \(n_A=5p\)：

- 一次测试大约排除 \(1/5\) 口令。
- 字典大小 \(10^6\)，剩余比例约 \((4/5)^t\)。

计算：

```text
10^6 * (4/5)^t <= 1
t >= log(10^-6) / log(4/5) ≈ 62
```

所以平均约 62 次成功测试会话确定唯一口令。

防御：

- 禁止模数含小素因子。
- 限制总假冒会话和连续失败次数。
- 不直接拒绝，返回随机值，消除 oracle。

## 6. 考场答题提醒

- 协议分析题一定写“攻击流程 + 违反性质 + 改进方案”。
- 口令协议题看到“服务器提前返回证明值/错误信息/拒绝信息”，优先想离线猜测或 oracle。
- 密钥交换题看到身份没进 Hash/签名/KDF，优先想 UKS 或 MITM。
- nonce 题不要只写“有随机数所以安全”，要看旧密钥泄露后是否能重放。
- 计算题所有步骤都写 mod p，Shamir 不要忘记常数项就是秘密。
- 选择题不会很难，但要分清消息认证、身份认证、完整性、新鲜性。
